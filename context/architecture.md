# Architecture Context

## Stack

| Layer     | Technology                           | Role |
| --------- | ------------------------------------ | ---- |
| Framework | Next.js 14 (App Router) + TypeScript | Frontend & API Routing |
| UI        | Tailwind CSS + shadcn/ui             | Styling & Component Primitives |
| Motion    | Framer Motion                        | High-quality page transitions |
| Content   | MDX                                  | Blog engine / Engineering insights |
| Database  | PostgreSQL (Neon.tech) + Prisma ORM  | Contact inquiries, blog metadata |
| Emails    | Resend                               | Transactional email delivery |
| Hosting   | Vercel                               | Edge delivery, CI/CD |

## System Boundaries
- `app/` - All Next.js pages, layouts, and API routes (`/api/contact`, `/api/blog`).
- `components/ui/` - Base UI primitives generated by shadcn/ui.
- `components/sections/` - Composable, domain-specific layout segments (e.g., `HeroSection`, `TechPhilosophy`).
- `content/blog/` - Static MDX files for engineering insights.
- `prisma/` - Schema definition (`schema.prisma`) and migrations.

## Storage Model
- **PostgreSQL**: Stores structured metadata, such as `ContactInquiry`, `BlogPost` metadata, and simple `PageView` records.
- **File System**: Blog posts are treated as code (`.mdx`).

## Invariants
1. We use PostgreSQL, not MongoDB, as the default database.
2. No complex global client state (Redux). Keep state in URL or leverage React Server Components.
3. Third-party scripts must not block rendering or drop Lighthouse scores below 95.
